## 📱 What It Is

slow hour is a daily tarot web app that gives users one card per day with a personalised "what this means for you" reading. The reading is generated by Claude AI using:

- The drawn tarot card and whether it's reversed
- Real-time planetary transits calculated against the user's natal chart
- The user's zodiac sun sign
- Memory of past readings (patterns, recent cards)

Users can also browse their year in the **year view** — a calendar grid showing every card drawn as a postage stamp, with the full reading accessible in a drawer.

## 🔬 How the Reading Engine Works

This is the full technical breakdown of how slow hour generates a personalised reading — from card draw through to the final sentence Claude writes. Everything here is intentional.

---

### 1. Card Draw

**File:** `app/page.tsx` → `loadTodaysCard()`

The card draw is completely client-side. No API call is made for this step.

**Flow:**
1. On load, check `localStorage` for `card-{YYYY-MM-DD}` and `reversed-{YYYY-MM-DD}`
2. If found → use the stored card. The user's card for the day is already committed.
3. If not found → draw fresh:
   - `Math.floor(Math.random() * 78)` picks from the full deck
   - `Math.random() < 0.30` decides reversal (30% chance)
   - Both are stored to `localStorage` immediately
4. The card stays identical for the rest of the day regardless of refreshes or re-opens

**Why truly random (not seeded):**
The original design used a deterministic hash of `date + birthdate` to derive the card. This meant testing with the same birth data always produced the same card — and users with similar birth dates would cluster around the same cards. Real tarot is not predictable. Switching to `Math.random()` committed immediately to localStorage gives the genuine "pulling from a shuffled deck" feeling, while still being stable for the whole day.

**Why 30% reversed:**
Traditional tarot readers vary widely (some never reverse, some do 50/50). 30% is close to the commonly used "cut the deck with your non-dominant hand" probability — it means most days the card reads upright, but reversed energy appears often enough to matter and feel significant when it does.

---

### 2. Natal Chart Calculation

**File:** `lib/utils/astrology-calculator.ts` → `calculateNatalChart()`
**Library:** `astronomy-engine` (Swiss Ephemeris-compatible, runs entirely server-side)

**What it calculates:**
- Planetary positions at birth (ecliptic longitude in degrees, 0–360°) for: Sun, Moon, Mercury, Venus, Mars, Jupiter, Saturn, Uranus, Neptune, Pluto
- House cusps using the **Equal House system** (ascendant + 30° increments)
- Ascendant derived from Local Sidereal Time at the birth location

**Location handling:**
- If birth location is provided → geocoded via **Nominatim (OpenStreetMap)** to lat/lon. Free, no API key required.
- If not provided → defaults to 0°N 0°E (equatorial). Claude is told this explicitly.

**Time handling:**
- If birth time is provided → used directly
- If not provided → defaults to 12:00 noon. Claude is told this explicitly.

**Why Equal Houses (not Placidus):**
Placidus (the most common house system in Western astrology) requires iterative calculations that break down at extreme latitudes and add significant complexity. Equal houses give meaningful house placements for every birth location without edge cases, and the difference between systems only becomes significant with a precise birth time — which is optional data here.

**Why Astronomy Engine:**
It's open source, runs server-side in Node.js with no external API calls, and uses the same VSOP87 ephemeris data as professional astrology software. Positions are accurate to well within the orbs we use.

---

### 3. Transit Detection

**File:** `lib/utils/astrology-calculator.ts` → `calculateActiveTransits()`

Transits are the real-time positions of planets in the sky right now, measured against the user's natal planet positions.

**Transiting planets checked (the "sky now" side):**
| Planet | Why included |
|---|---|
| Saturn | Structure, limitation, reality checks. Slow-moving. High impact. |
| Jupiter | Expansion, opportunity, meaning. Slow-moving. High impact. |
| Uranus | Sudden change, disruption, authenticity. Generational but personally triggering. |
| Neptune | Dissolution, intuition, illusion. Slow but psychologically deep. |
| Pluto | Transformation, power, the unavoidable. Generational; most intense when personal. |
| Mars | Drive, urgency, conflict. Fast-moving. Activates situations. |

**Natal planets checked (the "birth chart" side):**
Sun, Moon, Mercury, Venus, Mars, Jupiter

Inner natal planets are the personal points — they represent the user's core nature, emotional patterns, mind, relationships, and drive. Transits to these feel personal and immediate.

**Aspects detected:**
| Aspect | Exact Angle | Orb Used | Energy |
|---|---|---|---|
| Conjunction | 0° | ±10° | Fusion — energies merge and intensify |
| Opposition | 180° | ±10° | Polarity — tension between two sides |
| Square | 90° | ±8° | Friction — demands action or resolution |
| Trine | 120° | ±8° | Flow — natural ease and support |
| Sextile | 60° | ±6° | Opportunity — gentle opening |

**Why these orbs:**
Orbs define how close to exact an aspect needs to be to "count". Tight orbs (3–5°) are used in precise chart readings. Wider orbs (8–10°) are used here intentionally — daily readings work with the broad energetic weather, not only the most exact hits. Wider orbs also ensure we almost always find at least one active transit even for users who gave minimal birth data.

**Transit phase:**
| Orb from exact | Phase |
|---|---|
| < 1° | peak — exact and at full strength right now |
| 1°–3° | approaching — building in intensity |
| > 3° | beginning — just starting to form |

All detected transits are sorted by orb (tightest first) before the dominant transit selection step.

### 4. Dominant Transit Selection (Affinity Weighting)

**File:** `lib/utils/astrology-calculator.ts` → `getDominantTransit()`
**Data:** `lib/data/insight-structure-templates.ts` → `cardPlanetAffinity`

This is the step that connects the drawn card to the most thematically resonant transit currently active.

**The problem it solves:**
On any given day, there might be 6–15 active transits. Simply picking the tightest orb would ignore whether that transit is actually meaningful for the card drawn. The Ten of Swords (pluto/saturn energy) doesn't need Jupiter trine Moon even if it's the closest aspect — it needs the heavy, transformational transits.

**How it works:**
Each of the 78 cards has a pre-defined affinity profile:
```
'major-13': { planets: ['pluto', 'saturn'], aspects: ['square', 'opposition', 'conjunction'] }
// Death: weighted toward transformation planets, challenging aspects

'cups-3':   { planets: ['venus', 'jupiter'], aspects: ['trine', 'sextile', 'conjunction'] }
// Three of Cups: weighted toward pleasure/expansion planets, harmonious aspects
```

**Weighting formula:**
```
base weight = max(1, 10 - orb)          // tighter orb = higher base
planet rank bonus = (list_length - rank) × 6   // first planet = strongest bonus
aspect match bonus = +3                  // if aspect type is in affinity list
noise = sin(seed × 1000 + i) × 1.5     // small deterministic variation
```

The weights are used for weighted random selection — not a hard pick. This means:
- The affinity match wins most of the time
- But a non-affinity transit can occasionally surface if it's very tight
- The same card drawn on different days can select different transits as the sky shifts

**Why not just pick the highest weight deterministically:**
Deterministic selection with the same card would always pick the same transit as long as the transits were similar. The weighted random with the drawn card's ID as a noise seed means readings feel varied even when similar transits are active.

### 5. Claude Insight Generation

**File:** `app/api/calculate-transit/route.ts` → `generateClaudeInsight()`
**Model:** `claude-haiku-4-5-20251001`
**Max tokens:** 500
**Cost per call:** ~$0.0003

Claude receives a single structured prompt and returns a JSON object with four fields.

**What Claude is given:**
- The card name and orientation (upright/reversed)
- The card's core themes and emotional tone (from `card-archetypes.ts`)
- The dominant transit: transiting planet, natal planet, aspect type, phase, house activated
- Human-readable descriptions of what each planet and aspect means
- The user's sun sign (always)
- Data quality flags: whether birth time and location were provided
- Memory context: up to 10 pattern notes + 30 recent card names from past readings

**The four output fields:**

| Field | Format | Purpose |
|---|---|---|
| `keyPhrase` | 3–6 words, lowercase | The headline for today — specific to the collision of card + transit |
| `insight` | 2–3 sentences | What the combination actually means for this person right now |
| `action` | 1–2 sentences | A noticing practice — something to observe, not a task to complete |
| `memoryNote` | 1 sentence | A private pattern observation for future readings |

**Voice rules baked into the prompt:**
- Entirely lowercase
- No em dashes
- No questions at the end (the card already asks questions)
- No therapy-speak: no "journey", "navigate", "embrace", "explore", "tap into", "delve", "honor your", "give yourself permission"
- No filler affirmations — specific and real only
- Don't explain what the card means (the user can read that themselves)
- Don't explain what the transit means in general — say what it means for *them* right now
- Always weave in the sun sign when it's relevant

**Why Haiku and not Sonnet:**
The reading is structured JSON with 500 tokens max. Haiku handles this format reliably at a fraction of the cost. The quality difference between models at this output length and specificity is negligible — the prompt engineering does the heavy lifting. At scale (~1,000 users/day), Haiku costs ~$0.30/day vs ~$3+/day for Sonnet.

**Why JSON output:**
The four fields serve distinct UI roles. Returning JSON lets the app render each part exactly where it belongs — key phrase as the heading, insight as body copy, action in a separate visual block, and the memory note silently stored without being shown to the user.

### 6. Memory System

**File:** `app/components/TarotCard.tsx` + `app/api/calculate-transit/route.ts`

Each time a reading is generated, Claude produces a `memoryNote` — a one-sentence private observation about what's psychologically active for this person. These notes accumulate in localStorage.

**What's passed to Claude:**
- Up to **10 memory notes** (the most recent pattern observations)
- Up to **30 recent card names** (the last 30 cards drawn)

**How Claude uses it:**
The memory context comes with explicit instructions: *use this only to deepen your understanding of their current arc. Do not reference it directly. Do not say "as I've noticed" or anything that signals you remember past readings. Let it quietly inform your accuracy and sensitivity.*

The goal is invisible personalisation — readings that feel mysteriously accurate, not a chatbot recapping your history back at you.

**Memory note format:**
Notes are written as pattern observations, not events:
> "themes of self-worth and external validation are coexisting with a drive to act independently"

No card names, no planet names, no "you" — just the theme. This keeps notes useful across many future readings without becoming stale.

### 7. Data Quality Degradation

When users don't provide birth time or birth location, the reading adapts gracefully instead of failing or becoming generic.

| Data provided | What changes |
|---|---|
| Full data (date + time + location) | Full natal chart, precise house placements, accurate ascendant |
| Date + time only | House positions estimated (location defaults to 0°N 0°E) |
| Date + location only | House positions estimated (birth time defaults to noon) |
| Date only | Houses and rising sign estimated. Claude weights card themes + sun sign more heavily. |

**The prompt instruction when data is partial:**
> *"Birth time and location were not provided. The house and rising sign data is estimated. Weight your insight on the card themes and sun sign, using the transit as general planetary weather rather than a precise personal placement."*

The sun sign is always accurate — it only requires birth month and day. So even with zero birth time or location, Claude still has: the drawn card, the card's themes, the user's sun sign, the active planetary weather, and any accumulated memory notes.

---

### 8. Caching

Insights are cached in localStorage by `cardId + date` key:
```
insight-{cardId}-{YYYY-MM-DD}
```

Once generated, the insight is never re-fetched for that card+date combination. This means:
- Today's reading: generated once, cached forever
- Past card readings (year view): load instantly from cache with no API call
- Rate limiting is effectively per-day per-user since the cache prevents repeat calls

The `/api/calculate-transit` endpoint has additional rate limiting at the infrastructure level: **5 requests per IP per 24 hours** via Upstash Redis middleware — protecting against both abuse and accidental re-generation loops.

### 9. Full Request Flow (Single Reading)

```
User taps "what does this mean for me"
    ↓
TarotCard.tsx checks localStorage for cached insight
    ↓ (not cached)
POST /api/calculate-transit
    {birthDate, birthTime, birthLocation, cardId, isReversed, memoryNotes, recentCards}
    ↓
Middleware checks rate limit (Upstash Redis)
    ↓ (allowed)
Geocode birth location via Nominatim (OpenStreetMap)
    ↓
calculateNatalChart() — Astronomy Engine positions at birth
    ↓
calculateActiveTransits() — current sky vs natal chart
    → checks 6 transiting planets × 6 natal planets = up to 36 pairs
    → filters to those within orb of the 5 major aspects
    → sorts by orb (tightest first)
    ↓
getDominantTransit() — affinity-weighted selection
    → applies card-planet affinity scores
    → weighted random picks the transit
    ↓
generateClaudeInsight() — Claude Haiku call
    → prompt includes: card, transit, sun sign, memory context, data quality flags
    → returns JSON: {keyPhrase, insight, action, memoryNote}
    ↓
Response returned to client
    ↓
TarotCard.tsx renders insight, saves to localStorage cache
    ↓
memoryNote stored to localStorage for future readings
```

## 📝 Notes

- All user data is **localStorage only** — no server-side storage currently. Data is lost if user clears browser or switches device.
- The `SLOW_HOUR_ANTHROPIC_KEY` env var name is intentional (avoids conflicts with default `ANTHROPIC_API_KEY`).
- Card draw: truly random via `Math.random()`, stored to localStorage immediately. Consistent for the whole day once drawn. Different every day.
- Reversed cards: 30% probability, randomly assigned at first draw for the day.
- Birth time defaults to noon, birth location defaults to equator (0,0 UTC) when not provided — Claude is told when this is the case and adjusts accordingly.
- The year view stamp mask uses CSS `mask` shorthand (NOT `maskImage`) — 5 radial gradient layers with `mask-composite: subtract`.
